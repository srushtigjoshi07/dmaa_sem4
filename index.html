📌 1. Title

"Hybrid Telemetry Harvester: A SQL–MongoDB Production Monitoring System for Industrial Analytics and Downtime Prediction"

📄 2. Abstract

This project presents a dual-database architecture for production monitoring. Structured machine production data (shift outputs, good/scrap units, pressure metrics) is stored in a relational SQL database, while real-time sensor fault logs and schemaless telemetry drift signals are ingested into MongoDB. A web-based dashboard visualizes key performance indicators (KPIs), live sensor telemetry, and downtime forecasts. The system enables industrial analytics, active alarm tracking, and breakdown risk assessment, achieving a Global OEE Index of 98.52% in the deployed prototype.

⚠️ 3. Problem Statement

Modern industrial environments generate heterogeneous data: structured shift reports and semi-structured sensor fault logs. Storing both in a single database type leads to inefficiency—relational systems struggle with variable schemas (fault signatures), while document stores lack robust aggregation for production metrics.


Key Challenges:

  • Store structured production history in SQL (good units, scrap, piston pressure)
  • Store real-time sensor fault logs in MongoDB (variable fields, component-wise alerts)
  • Provide a unified dashboard for downtime prediction and operational visibility

🎯 4. Objectives

  • Design and implement a dual-database connector that routes SCADA telemetry to SQL and MongoDB
  • Build a web dashboard showing total relational logs, document logs, active alarms, and Global OEE Index
  • Implement Downtime Forecast Analyzer with breakdown risk percentage and cutoff window
  • Create Live Sensor Telemetry Rig with manual override capability
  • Display SQL table (production_history) and MongoDB collection (fault_logs)

⚙️ 8. Working Methodology

1. Data Ingestion: Simulated SCADA clusters generate operational metrics (core temp, vibration noise, oil pressure) and fault signals.
2. Hybrid Routing: Structured data → SQL (production_history), Unstructured fault logs → MongoDB (fault_logs).
3. Processing: Dashboard queries both databases in real-time via REST APIs.
4. Analytics: Global OEE Index computed from good vs. scrap units. Downtime Forecast based on historical fault patterns.
5. Visualization: Dashboard renders live telemetry gauges, SQL table, MongoDB fault list, and engineering recommendations.

📝 11. Algorithm

  1. SCADA Data Capture: Read sensor triple (CoreTemp, VibNoise, OilPressure) and shift production values (GoodUnits, ScrapUnits, PistonPressure)
  2. Structured SQL Insert: INSERT INTO production_history (good_units, piston_pressure_psi, scrap_units) VALUES (good, pressure, scrap);
  3. Semi-structured MongoDB Insert: Construct BSON document with timestamp, component, values, active_alarm flag → Insert into fault_logs
  4. OEE Index Calculation: OEE = (Total Good Units) / (Total Good + Total Scrap) × 100
  5. Breakdown Risk Forecast: Risk% = f(fault_frequency, scrap_trend, pressure_variance) → Output: 37%
  6. Cutoff Window: Based on MTBF extrapolation → ~140 hours
  7. Active Alarm Monitor: Count documents where active_alarm == true
  8. Dashboard Refresh: Update KPI cards, render SQL table, list MongoDB fault logs
