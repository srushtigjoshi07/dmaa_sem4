TITLE

FULL CONCEPT

Hybrid Telemetry Harvester
A SQL–MongoDB production monitoring system for real‑time industrial analytics and downtime prediction.
02

ABSTRACT

This work builds a dual‑database pipeline where structured shift data (good units, scrap, pressure) lives in SQL, while semi‑structured sensor faults and drift signals flow into MongoDB. A live dashboard — currently deployed — visualises OEE, active alarms, breakdown risk and real‑time telemetry. The system achieves 98.52% Global OEE and predicts downtime windows with ~140h cutoff, bridging the gap between relational integrity and document flexibility.

03

PROBLEM STATEMENT

⚙️
SQL RIGIDITY — struggles with schemaless fault signatures
📄
MONGODB ISOLATION — lacks robust production aggregation

Factories generate mixed data: structured shift logs and evolving sensor faults. No single database handles both efficiently. The result: fragmented visibility, delayed downtime prediction, and manual correlation.

WORKING METHODOLOGY

1 INGEST SCADA telemetry (core temp, vibration, oil pressure) & shift counters.
2 ROUTE structured metrics → SQL (production_history).
3 ROUTE fault logs & drift signals → MongoDB (fault_logs).
4 COMPUTE OEE, breakdown risk (37%), waste factor (2.7%).
5 RENDER live sensor rig & alarms; manual override triggers hybrid insert.

09

ALGORITHM

1. CAPTURE (CoreTemp, VibNoise, OilPressure, GoodUnits, Scrap)
2. SQL: INSERT INTO production_history (good_units, piston_pressure, scrap_units)
3. MONGODB: build { timestamp, component, drift, active_alarm } → fault_logs
4. OEE = total_good / (total_good + total_scrap) * 100
5. RISK = f(fault_freq, scrap_trend, pressure_var) → 37%
6. CUTOFF = MTBF extrapolation → ~140 hrs
7. ACTIVE_ALARMS = count(active == true) → 4
8. REFRESH DASHBOARD
